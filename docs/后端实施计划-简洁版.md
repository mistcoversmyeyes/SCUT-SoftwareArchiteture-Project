# OCR后端实施计划（简洁版）

## 实施阶段

### 第一阶段：基础框架搭建（2天）

**1. Go网关服务**
- 使用Gin框架搭建HTTP服务器
- 实现gRPC客户端连接Python服务
- 添加请求路由和中间件（CORS、Metrics、Logger）

**2. Python推理服务**
- 使用gRPC搭建推理服务器
- 集成ONNX Runtime作为推理引擎
- 实现模型加载和warmup机制

**3. Protobuf协议定义**
- 定义OCRRequest/OCRResponse消息
- 支持三种计算模式（SERVER_ONLY/CLIENT_PROVIDED/COMPARE_MODE）

### 第二阶段：模型集成（3天）

**1. 文本检测模型**
- 部署DBNet-resnet18（12M参数，ONNX格式）
- 实现单尺度和多尺度检测模式
- 目标：单张640x640图片检测 < 30ms

**2. 文本识别模型**
- 部署CRNN-MobileNetV3（5.2M参数）
- 实现批处理优化（动态batch）
- 目标：单行文字识别 < 10ms

**3. 模型优化**（可选）
- INT8量化（速度提升2-3倍）
- 如有GPU：TensorRT优化

### 第三阶段：性能优化（2天）

**1. 批处理队列**
- 实现智能批处理（1个直接推理，2-4个等待50ms，>4个直接分批）
- 避免头部阻塞问题

**2. 缓存机制**
- Redis缓存重复请求
- 内存缓存热点数据

**3. 并发控制**
- 限制最大并发推理数
- 实现请求队列和超时控制

### 第四阶段：实验数据收集（1天）

**1. 指标埋点**
- 检测耗时、识别耗时、总耗时
- 不同计算模式的对比数据
- 网络传输耗时（与前端配合）

**2. 数据持久化**
- CSV格式实时写入
- 支持后续分析和可视化

**3. A/B测试支持**
- COMPARE_MODE同时运行两种方案
- 自动生成对比报告

## 实验重点

1. **计算分区对比**：全服务端 vs 端侧检测+服务端识别 vs 全端侧
2. **网络条件影响**：WiFi/4G/5G下的延迟差异
3. **批处理效果**：单请求 vs 批请求的吞吐量对比
4. **模型精度权衡**：快速模式 vs 精确模式

## 核心原则

1. **不追求完美，追求可控** - 稳定比先进更重要
2. **测量一切** - 没有数据就没有优化
3. **为PA服务** - 展示计算分区效果是核心目标
4. **简单至上** - 能用10行代码解决的，不要写100行
