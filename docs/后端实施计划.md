# OCR后端实施计划

> 基于Linus Torvalds式架构分析制定的实施方案
>
> **核心原则**：满足PA作业要求，展示计算分区效果，保证1-2秒用户体验

## 一、架构总览

### 1.1 技术栈选择理由

| 组件 | 技术选择 | 选择理由 |
|------|----------|----------|
| API网关 | Go + Gin | 高并发、低延迟、天然适合网关场景 |
| 推理服务 | Python + gRPC | 深度学习生态成熟、模型部署简单 |
| 推理引擎 | ONNX Runtime | 跨框架兼容、部署友好、性能优秀 |
| 通信协议 | gRPC + Protobuf | 强类型、高性能、支持流式传输 |
| 模型格式 | ONNX | 统一格式、避免框架锁定 |

### 1.2 数据流设计

```
客户端请求 → Go网关（路由/鉴权/限流） → Python推理服务（模型推理） → 响应聚合 → 返回客户端
     ↓                                          ↓
  指标收集                                   性能监控
     ↓                                          ↓
  实验数据CSV                              Prometheus/日志
```

### 1.3 核心设计亮点

- **支持三种计算模式**：满足PA的计算分区实验要求
  - SERVER_ONLY: 全服务端计算
  - CLIENT_PROVIDED: 使用客户端检测结果
  - COMPARE_MODE: 同时运行两种方案对比

- **分层明确**：Go处理网络层，Python处理计算层，职责清晰
- **可观测性强**：每个阶段都有详细的性能指标

### 第二阶段：模型集成（3天）

#### Day 3-4: 文本检测模型

**目标**：集成DBNet模型，实现文本区域检测

**模型选择**：
- 主模型：DBNet-resnet18 (12M参数)
- 备选：DBNet-mobilenetv3 (更轻量)
- 来源：PaddleOCR预训练模型转ONNX

**实现要点**：
```python
class DBNetDetector:
    def __init__(self, model_path: str):
        self.session = onnxruntime.InferenceSession(model_path)
        self.input_size = (640, 640)
        self.threshold = 0.3
        self.box_threshold = 0.5

    def detect(self, image: np.ndarray, mode: str = "fast"):
        # 预处理：resize、normalize
        processed = self._preprocess(image)

        # 推理
        outputs = self.session.run(None, {
            self.session.get_inputs()[0].name: processed
        })

        # 后处理：二值化、轮廓提取、NMS
        boxes = self._postprocess(outputs[0], image.shape)
        return boxes
```

**性能目标**：
- 640×640图片：< 30ms
- 1920×1080图片：< 100ms（多尺度）

**优化策略**：

class BatchProcessor:
    def __init__(self, max_batch_size=8, max_wait_ms=50):
        self.queue = Queue()
        self.max_batch_size = max_batch_size
        self.max_wait_ms = max_wait_ms

    def process(self, items):
        # 根据数量动态调整策略
        if len(items) == 1:
            # 单个请求直接处理
            return self._process_single(items[0])
        elif len(items) <= self.max_batch_size:
            # 等待凑批
            return self._process_with_wait(items)
        else:
            # 拆分多批
            return self._process_multi_batch(items)

### 3.2 性能测试脚本

```python
# benchmarks/stress_test.py
import asyncio
import aiohttp
import time

async def test_latency(session, image_path, mode):
    start = time.time()
    async with session.post('/ocr',
        data={'image': open(image_path, 'rb'), 'mode': mode}) as resp:
        result = await resp.json()
    latency = (time.time() - start) * 1000
    return latency, result

async def benchmark(concurrent_requests=10, total_requests=100):
    results = []
    async with aiohttp.ClientSession() as session:
        for _ in range(total_requests // concurrent_requests):
            tasks = [test_latency(session, 'test.jpg', 'server_only')
                    for _ in range(concurrent_requests)]
            batch_results = await asyncio.gather(*tasks)
            results.extend(batch_results)

    # 分析结果
    latencies = [r[0] for r in results]
    print(f"P50: {np.percentile(latencies, 50):.2f}ms")
    print(f"P95: {np.percentile(latencies, 95):.2f}ms")
    print(f"P99: {np.percentile(latencies, 99):.2f}ms")
```

## 五、关键决策记录

### 5.1 为什么不用DeepSeek-OCR？

**Linus式分析结论**：
- DeepSeek-OCR是3B参数的端到端模型，无法拆分部署
- 使用它会导致无法完成PA的"计算分区"实验要求（占50分）
- 它需要16GB+ VRAM，部署成本高，不适合教学演示

### 5.2 为什么用Go+Python而不是纯Python？

**架构考虑**：
- Go网关层：处理高并发、低延迟的网络请求
- Python推理层：利用成熟的深度学习生态
- 分层清晰：便于实验不同的计算分区策略

### 5.3 为什么选择ONNX而不是PyTorch/TensorFlow？

**部署优势**：
- 跨框架兼容：可以用任何框架训练，统一用ONNX部署
- 性能优化：ONNX Runtime有成熟的优化
- 轻量部署：避免安装完整的深度学习框架
