## 1. 架构设计：数据结构优先

### 1.1 当前设计的数据流

```
Image → 端侧预处理 → [HTTP] → Go网关 → [gRPC] → Python推理 → [gRPC] → Go网关 → [HTTP] → 客户端
```

### 1.2 Linus式问题识别

**问题1：数据转换过多**
- 图片在HTTP → gRPC → ONNX之间转换3次
- **影响**：延迟增加、内存复制
- **缓解**：使用Protobuf Bytes类型，避免Base64编码

**问题2：三种计算模式导致分支爆炸**

❌ **糟糕的设计**（特殊情况处理）：
```python
if mode == "SERVER_ONLY":
    boxes = detector.detect(image)
    texts = recognizer.recognize(boxes)
elif mode == "CLIENT_PROVIDED":
    boxes = request.client_boxes
    texts = recognizer.recognize(boxes)
elif mode == "COMPARE_MODE":
    # 又跑一遍...这是浪费
```

✅ **好品味设计**（消除特殊情况）：
```python
class OCRPipeline:
    def process(self, image, boxes=None):
        # 统一接口：boxes为None时自动检测
        if boxes is None:
            boxes = self.detector.detect(image)
        texts = self.recognizer.recognize(image, boxes)
        return boxes, texts

# 使用时：
result = pipeline.process(image)              # SERVER_ONLY
result = pipeline.process(image, client_boxes) # CLIENT_PROVIDED
```

**好处**：
- 零if/else分支
- CLIENT_PROVIDED自然支持
- COMPARE_MODE删除（用AB测试工具离线对比）

### 1.3 Protobuf协议设计

```protobuf
// 好品味：数据结构清晰，职责单一
message OCRRequest {
  string request_id = 1;
  bytes image_data = 2;              // 原始图片
  repeated BoundingBox detections = 3; // 可选：客户端检测结果
}

message BoundingBox {
  repeated Point points = 1;         // 多边形顶点
  float confidence = 2;
}

message OCRResponse {
  string request_id = 1;
  repeated TextRegion results = 2;
  PerformanceMetrics metrics = 3;
}

message TextRegion {
  BoundingBox box = 1;
  string text = 2;
  float confidence = 3;
}
```

**设计理念**：
- 数据结构优先，而非代码逻辑优先
- `detections`为空 = SERVER_ONLY
- `detections`非空 = CLIENT_PROVIDED
- **无需mode字段**（数据结构已表达语义）

## 2. 技术栈选择：实用主义

### 2.1 为什么Go + Python而非纯Python？

#### Linus的质疑

> "这是在为架构而架构。对于一个教学项目，纯Python方案足够了。除非你能证明Go网关真的比FastAPI快10倍以上。"

#### 我们的反驳（求职导向）

| 维度 | 纯Python | Go + Python | 理由 |
|------|---------|------------|------|
| **开发速度** | ✅ 快 | ❌ 慢（需跨语言） | PA紧急时Python优先 |
| **简历价值** | ❌ 普通 | ✅ **微服务架构** ⭐ | 求职核心竞争力 |
| **技术深度** | ❌ 单一栈 | ✅ **gRPC + Protobuf** ⭐ | 展示分布式能力 |
| **职责分离** | ❌ 混合 | ✅ **API层 vs 推理层** | 架构理解 |
| **性能** | 🟡 够用 | ✅ 理论更好（但差异小） | 非核心因素 |

**最终结论**：
- **两阶段策略**：先Python MVP（Day 1-7），后Go升级（Day 8-14）
- **理由**：降低风险（确保PA完成）+ 最大化求职价值

### 2.2 为什么PyTorch而非ONNX？

#### Linus的审查

> "ONNX确实更适合生产部署。但模型转换是额外风险。对于教学项目，PyTorch直接加载可能更快启动。"

#### 风险-收益分析

| 方案 | 优势 | 风险 | 决策 |
|------|------|------|------|
| **PyTorch直接** | - 零转换风险<br>- 开发最快<br>- 调试简单 | - 性能略差<br>- 部署包大 | ✅ **Phase 1推荐** |
| **ONNX Runtime** | - 性能最优<br>- 跨框架<br>- 部署轻量 | - 转换可能失败<br>- 调试复杂 | 🟡 **Phase 2可选** |
| **Tesseract OCR** | - 简单稳定 | - 精度低<br>- 无深度学习 | ❌ 仅降级方案 |

**最终方案**：
1. **Day 0**：验证PaddleOCR（PyTorch）可行性
2. **Day 1-7**：PyTorch部署，快速跑通
3. **Day 11-13**（可选）：尝试ONNX转换，如失败无影响

## 3. 性能目标：现实检验

### 3.1 声称的目标 vs 现实

| 指标 | 文档目标 | 现实分析 | Linus评价 |
|------|---------|---------|-----------|
| 端到端延迟 | 1-2秒 | WiFi 10ms + 推理50ms + 网络往返？ | "网络延迟被低估了" |
| 检测速度 | <30ms | DBNet-ResNet18 CPU约50-100ms | "在哪测的30ms？" |
| 识别速度 | <10ms/行 | CRNN单行约10-20ms | "勉强合理" |

### 3.2 批处理的微优化陷阱

文档提到的批处理策略：
```
1个请求：直接推理
2-4个请求：等50ms凑批
>4个请求：分批处理
```

#### Linus的警告

> "如果网络延迟已经100ms，优化30ms推理时间有意义吗？这是过度设计的微优化。"

**问题分析**：
- 第2个请求到达时，第1个已完成，白等50ms
- 第5个请求到达时，前4个还在等，又要再等一轮
- **增加复杂度但收益不明**

**建议**：
- ✅ **MVP阶段**：简单队列，无批处理
- 🟡 **优化阶段**：如真的有性能瓶颈再加

## 4. 计算分区实验：核心目标对齐

### 4.1 PA的真实要求

> 展示不同计算分区对性能的影响，说明影响因素（网络、设备、输入数据）

### 4.2 当前方案分析

| 模式 | 定义 | 是否必需 | Linus评价 |
|------|------|---------|-----------|
| **SERVER_ONLY** | 全服务端计算 | ✅ 必需 | "基准方案，合理" |
| **CLIENT_PROVIDED** | 端侧检测+服务端识别 | ✅ 必需 | "展示分区，合理" |
| **COMPARE_MODE** | 同时运行两种方案 | ❌ 删除 | "伪需求！用AB测试工具" |

#### COMPARE_MODE为什么是伪需求？

> "没有用户会在生产环境同时运行两种方案。这是为了实验而实验，不是解决实际问题。"

**更好的方案**：
```python
# 离线对比工具（单独脚本）
def compare_modes(test_images):
    for img in test_images:
        result_server = run_server_only(img)
        result_client = run_client_provided(img)

        print(f"SERVER: {result_server.latency}ms")
        print(f"CLIENT: {result_client.latency}ms")
        print(f"Diff: {result_client.latency - result_server.latency}ms")
```

**好处**：
- 删除运行时复杂度
- 实验数据更准确（无相互干扰）
- 代码更简洁

## 5. 微服务架构：值得的复杂度

### 5.1 架构复杂度分析

| 层级 | 组件 | 技术栈 | 职责 |
|------|------|-------|------|
| **API网关** | Go服务 | Gin + gRPC Client | HTTP接入、限流、监控 ⭐ |
| **协议层** | Protobuf | .proto定义 | 跨语言通信 ⭐ |
| **推理服务** | Python服务 | gRPC Server + PyTorch | 模型推理 |
| **数据存储** | CSV/JSON | - | 性能数据（可选Redis） |

**总计**：2种语言 + 2种通信协议（HTTP + gRPC）

### 5.2 Linus的裁决

> "对于一个5-7人的教学项目，这是增加复杂度。FastAPI + Uvicorn完全能处理几百RPS。"

**备选方案决策树**：

```
└─ ❌ No  → 备选方案
           ├─ Plan A: HuggingFace ONNX模型
           ├─ Plan B: Tesseract OCR（降级）
           └─ Plan C: 在线API（最后手段）
```

### 6.2 次高风险：R2 性能目标

**Day 5基准测试**：
```python
# 性能基准脚本
def benchmark():
    images = load_test_images()  # 10张不同尺寸

    for img in images:
        start = time.time()
        result = ocr_pipeline(img)
        latency = time.time() - start

        print(f"Size: {img.size}, Latency: {latency:.2f}s")

    # 判定：90%图片 < 2秒？
```

**如果不达标**：
1. 图像预处理降分辨率（640→480）
2. 模型量化（FP32→INT8）
3. GPU加速（如有条件）
4. 调整目标为"分析影响因素"而非"保证<2秒"
